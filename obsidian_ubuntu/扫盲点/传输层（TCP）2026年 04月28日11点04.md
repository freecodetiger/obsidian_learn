TCP 协议是如何可靠的？
第一 ,序列号，每一次发送数据包都会带上序列号。，可以解决乱序问题。
确认应答号，用来解决丢包问题，确认当且仅当这个序号之前的所有数据都被接收。

控制位：
ACK：确认应答
RST：强制断开连接（出现异常）
SYN：希望建立连接
FIN：希望断开连接

网络层协议是不可靠的，它（IP 协议）不保证数据包的按顺序交付、也不保证数据的完整性。
正是 TCP 协议保证了网络包**无损坏、无间隔、按顺序、非冗余**的。

TCP 面向连接：一对一。
TCP 可靠：**序列号+重试+三次握手四次挥手**保证数据包一定能到达接收端。
TCP 基于字节流：接收方在收到重复的数据时会自动丢弃，在收到前一个数据包之前坚决不处理后一个数据包，保证数据有序。

建立 TCP 连接实际上就是达成三个信息的共识：
Socket：由 IP 地址和端口号组成
序列号：用来解决乱序问题。
窗口大小：拥塞处理（流量控制）

通过 Socket 的 IP 地址和端口号可以唯一标识一个连接。
每个 TCP 连接都是一个文件，会占用一个文件描述符。

UDP 不需要连接，同时支持一对多。
UDP 不在乎网络拥堵。
UDP 首部只有 8 字节，开销小。

三次握手为什么好（为什么需要三次握手）：
从三个方面：Socket、序列号、窗口大小。
三次握手可以阻止重复历史连接的初始化。
三次握手可以同步双方的初始序列号。
三次握手可以避免资源浪费。

三次握手最重要的就是避免旧的连接的干扰。
因为拥塞或者服务端宕机引发的客户端重试 SYN 报文2可能 慢于最初发的 SYN 报文 1，服务端接收到旧的报文 1 会基于旧的序列号+1 发送 SYN+ACK 报文给客户端，随后服务端进入接收数据状态（单方面建立连接），客户端会检查这次的报文序列号是不是和最新发出去的 SYN 报文逻辑一致，结果不一致，客户端就会发 RST 报文，强制结束连接，服务端收到后就会终止连接。随后新的 SYN 报文 2 到达服务端，就会正常建立 TCP 连接，这个 SYN 报文 1 就是历史连接。

第三次握手就是给了客户端一次一票否决权，让客户端精准否定旧的幽灵连接。
TCP 三次握手建立连接最主要原因就是防止历史连接初始化连接。

三次握手是保证同步双方初始序列号的最少握手次数，两次不够，四次太多余。
第一次客户端发 SYN，服务端收到后知道了客户端的初始序列号，
第二次服务端发 ACK+SYN，客户端收到后，知道了客户端的初始序列号，
但是客户端不知道有没有收到自己的初始序列号（需要 ACK）
第三次客户端发 ACK，服务端收到后彻底放心，TCP 连接建立。

TCP 连接其实很昂贵，一旦建立就会持续占有内存和文件描述符还有接收、发送缓冲区等物理内存空间。所以三次握手的第三次握手从这个角度出发可以大大节约内存资源。如果只有两次握手，一旦服务器收到 syn 就立马开始建立建立连接，那么如果因为网络卡顿，客户端发了 100 个 syn，服务端就会建立 100 个连接，但是其实前面 99 个都不用处理，第三次握手就是用来处理幽灵连接的建立的。
当服务端收到 syn 时，只会进入一个半连接状态，只会分配极少的内存，不分配昂贵的缓冲区，只有收到客户端的 ack 时，才会分配完整的连接资源，正式建立 TCP 连接。

如果第三次握手丢失了，服务端不会傻等，而是采用指数退避重发 SYN+ACK（避免网络更加拥堵）
客户端收到后，会立马补发 ACK。
如果指数退避次数达到上限，服务端会直接断开半连接，释放资源。
那么客户端不知道连接没有被建立成功，在发送第一个数据包的时候，客户端会认为这个数据包很莫名其妙，于是向客户端回复一个 RST 报文，强制关闭连接。
