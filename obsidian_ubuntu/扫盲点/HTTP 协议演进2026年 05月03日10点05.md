HTTP（超文本传输协议）自诞生以来，为了适应日益复杂的互联网需求，经历了多次重大的版本更迭。我们常说的“HTTP的种类”，实际上指的是 **HTTP 协议的不同版本**。

目前互联网上最主要的历史版本和现行版本有五种：HTTP/0.9、HTTP/1.0、HTTP/1.1、HTTP/2 和 HTTP/3。

以下是它们的演进历史以及核心区别：

### 1. HTTP/0.9 (1991年) —— 远古原型

这是 HTTP 的第一个草案版本，非常简陋。

- **特点：** 极其简单，**只支持 GET 方法**。
    
- **区别与限制：** 没有请求头（Headers），没有状态码（Status Codes），服务器只能响应纯 HTML 文本，发送完毕后立即断开 TCP 连接。
    

### 2. HTTP/1.0 (1996年) —— 奠定基础

随着网页内容的丰富（图片、音频出现），HTTP 需要传输更多类型的数据。

- **特点：** 引入了**请求头和响应头**（Headers）、**状态码**（如 200、404）以及 `POST` 和 `HEAD` 方法。
    
- **区别与限制：**
    
    - **短连接（核心痛点）：** 每次发送请求都必须经历完整的 TCP 三次握手，请求结束后立即断开连接。这在加载包含大量图片和资源的网页时，会导致极大的网络延迟和资源消耗。
        
    - 存在**队头阻塞**（Head-of-Line Blocking）问题：前一个请求没有收到响应，后一个请求就不能发送。
        

### 3. HTTP/1.1 (1997年) —— 经典主流

这是互联网上使用时间最长、目前依然被广泛支持的版本。它主要为了解决 HTTP/1.0 的性能瓶颈。

- **特点与核心改进：**
    
    - **长连接 (Keep-Alive)：** 默认开启长连接（`Connection: keep-alive`），允许在一个 TCP 连接上发送多个 HTTP 请求和响应，大幅减少了 TCP 握手的开销。
        
    - **管道化 (Pipelining)：** 允许客户端在收到前一个响应之前连续发送多个请求。但由于服务器依然需要按顺序返回响应，如果第一个响应很慢，后面的响应依然会被阻塞（**依然存在 HTTP 层的队头阻塞**），因此实际应用中很少开启。
        
    - **引入 Host 头：** 允许一台物理服务器基于不同域名托管多个网站（虚拟主机技术）。
        
    - **断点续传：** 支持 `Range` 头，允许只请求资源的一部分。
        

### 4. HTTP/2 (2015年) —— 性能飞跃

随着现代网页越来越复杂，单个页面可能需要请求几十上百个资源，HTTP/1.1 显得力不从心。HTTP/2 是对底层传输机制的全面重构，但保持了 HTTP/1.1 的语义（方法、状态码等不变）。

- **特点与核心改进：**
    
    - **二进制分帧：** 数据在传输前会被拆分为更小的二进制帧，而不是 HTTP/1.x 的纯文本格式，解析更高效。
        
    - **多路复用 (Multiplexing)：** **这是 HTTP/2 最大的突破。** 允许在一个 TCP 连接中并发交错地发送多个请求和响应，彻底解决了 HTTP 层的“队头阻塞”问题。浏览器不再需要为了并发请求而建立多个 TCP 连接。
        
    - **头部压缩 (HPACK)：** HTTP/1.x 中每次请求都会携带大量重复的 Headers（如 Cookie、User-Agent）。HTTP/2 使用 HPACK 算法压缩头部，并在通信双方维护头部字典，极大减少了冗余数据的传输。
        
    - **服务器推送 (Server Push)：** 服务器可以主动将客户端可能需要的资源推送到客户端缓存中，而无需等待客户端明确请求。
        

### 5. HTTP/3 (2022年) —— 颠覆底层协议

虽然 HTTP/2 解决了 HTTP 层的队头阻塞，但由于它依然基于 TCP，如果网络出现丢包，**TCP 层的队头阻塞**依然存在（丢掉的包会导致整个 TCP 连接暂停，等待重传）。HTTP/3 决定抛弃 TCP。

- **特点与核心改进：**
    
    - **基于 QUIC 协议 (底层使用 UDP)：** HTTP/3 将传输层协议从 TCP 换成了 Google 开发的 QUIC 协议（基于 UDP）。
        
    - **彻底解决队头阻塞：** QUIC 原生支持多路复用。由于基于 UDP，各个数据流之间相互独立，某一个流的数据丢包，只会影响该流自身，不会阻塞其他并发流。
        
    - **0-RTT 连接建立：** QUIC 将**加密握手和连接握手合并**，曾经连接过的服务器甚至可以实现 0 个往返时间（0-RTT）就开始发送数据，建立连接极快。
        
    - **连接迁移：** 移动设备在不同网络之间切换（例如从 Wi-Fi 切换到 5G）时，因为 QUIC 使用 Connection ID 而非 IP 端口四元组来识别连接，所以可以保持连接不断开，实现无缝切换。

### 核心区别总结 (对比表)

|**特性**|**HTTP/1.0**|**HTTP/1.1**|**HTTP/2**|**HTTP/3**|
|---|---|---|---|---|
|**发布年份**|1996|1997|2015|2022|
|**传输层协议**|TCP|TCP|TCP|**UDP (QUIC)**|
|**连接管理**|短连接 (每次建连)|默认长连接 (复用TCP)|单连接多路复用|QUIC连接 (支持连接迁移)|
|**队头阻塞问题**|极其严重|HTTP层队头阻塞|TCP层队头阻塞|**彻底解决**|
|**并发机制**|需建立多个TCP连接|需建立多个TCP连接|**多路复用** (单TCP并发)|QUIC原生多路复用|
|**数据格式**|纯文本|纯文本|**二进制帧**|二进制帧|
|**Header压缩**|无|无|**HPACK 算法**|**QPACK 算法**|
|**安全性**|配合 HTTPS|配合 HTTPS|配合 HTTPS (几乎强制)|**内置 TLS 1.3** (强制加密)|
http/2 解决了应用层的队头阻塞（http 队头阻塞）
但是不能解决传输层的队头阻塞（TCP 队头阻塞）
