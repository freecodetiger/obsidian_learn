UDP 协议没有连接的概念，也没有序列号和重传机制，所以很不可靠，怎么办？

给 UDP 在应用层增加 UDP 加上序列号和确认号，加上超时重传的定时器，加上滑动窗口进行流量控制，加上拥塞控制算法（探测网络带宽，避免把网络弄崩溃）

基于 UDP 的可靠传输协议，本质上就是在应用层重新实现了一遍 TCP 可靠机制。

在 TCP 连接中，由 ip 端口四元组唯一确定连接，如果用户切换网络类型（移动网络和 wifi），导致 ip 地址变化，TCP 不得不重新建立连接，而 QUIC 不认 IP 地址，它在握手时生成一个随机的 64 位连接 ID，ip 地址变化不影响，只要这个连接 id 不变就可以复用连接。

#### 极速握手 (0-RTT 与 1-RTT)

- **传统 TCP + HTTPS：** 需要先进行 TCP 三次握手（1-RTT），再进行 TLS 加密握手（1~2 RTT）。聊上天之前，数据包已经在路上空跑了 2 到 3 个来回。
    
- **QUIC 的解法：合并握手。** QUIC 内置了 TLS 1.3 加密。它将“连接建立”和“密钥交换”合并在一起，首次连接只需要 1 个 RTT。如果你以前和这个服务器连过，它甚至可以实现 **0-RTT**（把数据和握手包一起发过去），速度快到极致。**
